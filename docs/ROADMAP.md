# satmodel 路线图

当前 `satmodel` 的 `v0.1` 目标是形成清晰、可调用、可测试的单速率姿态仿真库。后续版本应沿用现有接口继续扩展，避免重新退回到单脚本耦合结构。

当前稳定研究库版本由 Git 标签 `v0.1.0-current` 保留；平台化迭代默认在 `codex/platform-v0.2` 分支推进。

## 路线原则

1. 保留 `ScenarioRunner`、`SimulationConfig`、`SimulationResult`、`build_default_system()` 和 `build_cubesat_reaction_wheel_system()` 等现有入口。
2. 平台能力优先加在编排层：场景配置、实验运行、结果归档、报告和验证，而不是把所有新逻辑塞进 `system.py`。
3. 第一阶段采用轻量可用路线：标准库优先，CSV、Markdown 和 JSON 优先，复杂依赖后置。
4. 每一阶段都要补充测试和验收准则，保证旧示例、旧 API 和核心物理路径继续可用。

## v0.2：轻量平台骨架

目标是把当前单场景仿真库升级为配置驱动的实验平台雏形。

计划交付：

1. 增加轻量 `ScenarioSpec` 规划入口，描述时间设置、初始状态、系统构造、控制器和输出。
2. 增加场景加载与编译边界，例如未来的 `load_scenario()` 和 `compile_scenario()`。
3. 增加通用 `StudyRunner` 规划入口，支持单场景运行、参数扫描和轻量 Monte Carlo seed 序列。
4. 增加结果归档规划：`manifest.json`、`metrics.csv`、`time_history.csv` 和 `README.md` 报告。
5. 明确旧 API 兼容策略：`ScenarioRunner.run(SimulationConfig)` 行为不变，新平台入口作为旁路增强。

暂不做：

- 不强制引入 Pydantic、Parquet、HDF5、数据库或 Web 服务。
- 不重写单速率仿真循环。
- 不改变现有示例脚本的调用方式。

## v0.3：多速率调度与任务模式

目标是让仿真能表达更接近飞控系统的时序和任务逻辑。

计划交付：

1. 增加多速率调度器，让动力学、传感器、估计器、控制器和日志记录可以使用不同采样周期。
2. 保留单速率语义基线，等频配置下应与 `ScenarioRunner` 结果保持可解释的一致性。
3. 增加任务模式管理，例如 detumble、惯性定向、对日定向、对地指向、安全模式和大角度机动。
4. 增加故障注入事件，例如反作用轮失效、传感器丢包、执行器降额和模式切换。
5. 将事件日志纳入结果产物，支持复现实验和故障回放。

## v0.4：高保真工程模型

目标是提高物理模型、传感器和执行机构的工程可信度。

计划交付：

1. 在 `BoxGeometry` 基础上增加面元几何，使气动、SRP 和磁扰动分析具备更高保真度。
2. 将盒体投影气动和太阳光压模型逐步替换为面元积分模型。
3. 扩展轨道和环境 provider，引入由星历驱动的太阳方向，并验证 IGRF、NRLMSIS 和 TLE/SGP4 适配器。
4. 在反作用轮模型中增加轴承摩擦、电机一阶滞后、安装误差和动量卸载逻辑。
5. 增加磁力矩器、星敏感器、太阳敏感器、磁强计和更丰富的陀螺误差模型。
6. 将物理验证扩展到 Monte Carlo 场景和 Basilisk、GMAT、Tudat 风格参考案例。

## v0.5：可视化、发布与产品化

目标是把实验平台变成更易用、更可发布、更适合团队协作的工具。

计划交付：

1. 增加三维姿态回放、轮组遥测曲线、扰动力矩预算和模式时间线可视化。
2. 增加 Markdown/HTML 报告生成器，后续可扩展到 Streamlit 或桌面界面。
3. 增加实验目录浏览、指标筛选和结果对比能力。
4. 增加更严格的包构建检查、安装后 smoke test、CI 流程和发布检查。
5. 补齐许可证文件、变更日志、版本化策略和结果 schema 迁移策略。

## 长期研究方向

1. 在固定步长 RK4 积分器之外，增加 `solve_ivp` 动力学后端。
2. 增加批量最小二乘、EKF 惯量辨识和完整惯量矩阵可观性诊断。
3. 建立控制器基准测试集，用统一场景比较 PD、LADRC 和后续控制器。
4. 将配置、数据产品和验证基准沉淀为可复现实验模板。
