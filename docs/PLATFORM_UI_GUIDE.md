# satmodel 平台界面说明

## 1. 界面目标

平台界面的目标不是把所有功能堆在一个页面里，而是把卫星姿态控制实验组织成一条清楚的工作流：

`选实验 -> 建计划 -> 跑实验 -> 看结果 -> 做对比 / 回放 -> 回到下一轮实验`

当前界面采用左侧导航 + 右侧工作台布局，重点强调三件事：

1. 页面切换清楚，不靠长滚动找入口。
2. 每个页面先给判断，再给细节。
3. 结果阅读、计划管理和实验主线保持一致。

当前平台还支持 URL hash 级别的工作台定位。也就是说刷新页面后会保留当前页签和子工作台，并且可以直接打开某个具体入口，例如结果总览里的风险结果视图，而不需要重新一层层切换。

## 2. 当前页面结构

### 平台总览

作用：先说明平台是什么、当前工作区里有什么、标准实验主线怎么走。

首屏工作台：

- `平台定位`
- `实验主线`
- `当前资产`

首页默认应压成一个短控制台，而不是多段长说明，至少固定以下几类块：

- `首页信号卡`：当前阶段、主线覆盖、资产状态、下一动作
- `首页五步入口`：直接跳到实验库、创建实验、计划管理和结果工作区
- `主线入口建议`：右侧固定给出“建议先跑 / 当前主线缺口 / 展示亮点”三张实验入口卡

首页中的五步主线卡片应直接可点击，而不只是静态说明。

适合：

- 第一次打开平台
- 给老师或同事介绍平台结构
- 判断当前工作区是否已经形成可展示资产

### 实验库

作用：从成熟实验入口出发，而不是从空白 JSON 开始。

当前分成四类视图：

- `实验主线`
- `实验地图`
- `示例入口`
- `模板库`

右侧详情区固定按三层阅读：

- `协议概览`
- `代表资产`
- `学术说明`

同时补一张固定的 `结果阅读任务书`，直接说明：

- 跑完后先看什么
- 再去哪里做对比或回放
- 报告结论应该抓什么
- 下一步接哪组实验

其中 `实验地图` 专门回答两类问题：

- 当前平台里到底有哪些成熟实验
- 每个实验属于哪条协议、当前有没有代表计划和代表结果

适合：

- 找到当前最应该做的实验
- 复用已有模板
- 沿标准主线继续派生下一组计划

### 创建实验

作用：把一个实验问题整理成配置化计划。

当前创建器按四步组织：

1. `研究问题`
2. `场景与变量`
3. `任务与验收`
4. `预览与生成`

页内固定工作台：

- `设计焦点`
- `实验规模`
- `结果出口`

同时固定三类设计卡：

- `标准实验协议卡`
- `变量设计任务书`
- `任务与验收任务书`
- `结果阅读任务书`

同时创建器驾驶舱和各阶段任务书应共享一条 `主线资产链`：

- `代表模板 -> 代表计划 -> 代表结果 -> 下一协议实验`

这样用户从实验库进入创建器后，不会丢掉“当前模板在整条主线里的位置”。

这样创建器不是单纯填字段，而是先把模板、变量、任务、验收和结果图包压成一条设计链。

创建器本身应采用分栏工作台：

- 左侧固定 `步骤切换 + 设计驾驶舱`
- 右侧只显示当前步骤内容

这样用户不会在创建器里来回长滚动，而是始终围绕当前阶段决策。

适合：

- 从场景生成单场景计划
- 做参数扫描
- 做 Monte Carlo
- 做任务模式切换和执行器边界实验

### 计划管理

作用：把实验计划当作资产管理，而不是只当作 JSON 编辑。

当前分成四个子页：

- `实验池`
- `当前计划`
- `版本与归档`
- `场景与记录`

其中 `实验池` 再分成两层：

- `工作台总览`
- `计划清单`

当前首屏采用固定双栏：

- 左侧 `实验选择`
- 右侧 `计划资产总览`

下方再进入统一的 `计划管理工作台`，避免一打开就是长表格或长 JSON。

其中左侧实验选择区应固定成侧栏式工作台，优先承载：

- 最近计划选择
- 打开实验库
- 进入创建实验
- 刷新计划池
- 焦点计划 / 结果状态 / 主线位置 / 下一动作这组小信号卡
- 当前焦点、当前缺口、展示入口这组三张主线建议卡

适合：

- 批量筛选和运行计划
- 编辑已有计划
- 复制、重命名、归档和恢复计划
- 维持实验版本链条清晰

当前 `计划管理` 页也固定成四层工作台：

1. `计划焦点`
2. `主线看板`
3. `资产主线`
4. `下一步动作`

其中 `下一步动作` 不应只给按钮，还应固定一组 `当前建议的实验补强方向` 卡片，直接指出：

- 当前焦点实验属于哪一环
- 现在优先补代表计划还是代表结果
- 如果要做展示，应从哪一环切入

同时右侧应固定一组共享语义卡，不因为进入计划管理就退回成“只看文件”：

- `当前计划实验说明卡`
- `当前计划结果阅读任务书`
- `当前计划主线资产链`

这样用户在计划管理里也能直接看到：

- 这份计划到底在回答什么研究问题
- 跑完后结果应先看什么
- 它和标准模板、代表结果、下一步实验是什么关系

其中 `计划焦点` 首屏优先回答三件事：

- 现在在推进哪个计划
- 这个计划有没有对应结果
- 下一步是编辑、运行，还是回结果层继续读图

其中 `当前计划` 子页本身也应继续压成切换式工作台，而不是把计划画像、结构概览、JSON、场景摘要和操作记录一次性全部摊开。推荐固定三块子工作台：

- `计划画像`
- `结构编辑`
- `场景与记录`

这样进入计划编辑器后，用户可以先判断这份计划在主线里的位置，再进入字段级修改，最后回到场景和操作记录做复查，不需要靠长滚动找信息。

其中 `主线看板` 专门负责：

- 按协议阶段检查当前工作区覆盖到哪一步
- 判断哪些协议已经有计划、哪些协议已经有结果
- 直接从计划管理回到某条协议的入口实验

### 结果总览

作用：先判断结果是否值得继续读，再进入细节。

当前分成三层：

- `驾驶舱`
- `结果列表`
- `当前结果`

其中 `驾驶舱` 首屏再固定成双栏：

- 左侧 `结果驾驶舱`
- 右侧 `结果入口板`

其中驾驶舱固定按三类判断组织：

- `焦点结果`
- `风险结果`
- `主线导航`

同时结果总览右侧入口板应固定共享三张语义卡：

- `焦点结果实验说明卡`
- `焦点结果阅读任务书`
- `焦点结果主线资产链`

这样用户在结果首页就能直接看到：

- 这组结果到底对应什么实验问题
- 应该先看摘要、诊断、图表、对比还是回放
- 它和代表计划、代表结果、下一步实验的关系

适合：

- 快速定位当前最重要的结果
- 先处理失败或边界结果
- 把结果重新接回标准实验主线

当前默认区分两类入口：

- `代表结果`：优先按主线内更稳定、更适合作为展示入口的结果选择
- `最近结果`：单独按时间排序，用于回看刚跑完的实验

当前结果浏览器继续细分为四类视角：

- `代表结果`
- `结果地图`
- `风险结果`
- `最近结果`

其中 `代表结果` 和 `最近结果` 浏览器都应共享语义卡，而不只是结果列表：

- `代表/最近结果实验说明卡`
- `代表/最近结果阅读任务书`
- `代表/最近结果主线资产链`

这样结果浏览器就不只是“挑一个结果”，而是先完成第一轮结果判断，再决定是否进入详情页。

其中 `结果地图` 专门负责：

- 按协议阶段检查哪些主线已经形成稳定代表结果
- 看结果覆盖缺口到底在基线、环境、任务还是执行器边界
- 直接从结果页回到某条协议的入口实验或代表计划

实验库、计划管理和结果层现在都应共享同一张 `主线资产链卡`：

- 代表模板
- 代表计划
- 代表结果
- 下一步动作

这样从实验库进入计划管理或结果页时，不需要重新判断“这条主线现在该看哪个计划、哪个结果、下一步去哪里”。

同时三页都应能展开同一套 `标准实验主线看板`：

- 当前主线在哪一环
- 这一环的代表计划是什么
- 这一环的代表结果是否已形成
- 下一环应该接哪组实验

### 结果对比

作用：比较关键 run 的差异。

当前页内工作台：

- `结论判断`
- `参数差异`
- `回放出口`

首屏结构：

- 上层先给 `当前对比组合`
- 中层给 `首屏判读台`
- 下层再进入 `任务书 / 曲线 / 指标卡`

当前首屏再固定成双栏：

- 左侧 `关键 Run 对比`
- 右侧 `对比入口板`

适合：

- 比较最佳 / 最差 run
- 比较不同参数点
- 比较环境、任务或执行器差异

### 姿态回放

作用：把结果从表格和曲线还原成动态过程。

当前页内工作台：

- `代表回放`
- `任务片段`
- `调度快照`

下方细读区：

- `动画`
- `任务时间线`
- `运行时调度`

首屏结构：

- 上层先给 `当前回放对象`
- 中层给 `动态结论`
- 再决定进入 `任务片段` 或 `调度快照`

当前首屏再固定成双栏：

- 左侧 `三维姿态回放`
- 右侧 `回放入口板`

适合：

- 展示姿态动态过程
- 复查模式切换
- 对照 mission timeline 与 runtime schedule

### Dashboard 预览

作用：预览实验目录中可独立打开的静态结果页面。

适合：

- 查看结果归档是否完整
- 演示单个实验目录
- 对外分享 `dashboard.html`

## 2.1 结果阅读固定骨架

当前 `当前结果` 页已经固定成两层：

1. 首屏驾驶舱：
   - `判定焦点`
   - `读图路线`
   - `推进动作`
2. 结果标签层：
   - `概要`
   - `诊断`
   - `图表`
   - `链路`
   - `产物`

其中 `图表` 标签下再固定成四包阅读：

- `图包 A 基线图包`
- `图包 B 诊断图包`
- `图包 C 边界图包`
- `图包 D 回放图包`

这样结果解读先共享统一骨架，再允许不同实验保留各自重点。

`概要` 标签内部也固定成两块：

1. `学术摘要首屏`
2. `结果摘要画布`

也就是说首屏不再把解释文字铺得很长，而是先用结论、验收、图包、下一实验和四步结果画布把信息压成一页。

当前结果页首屏还应固定两层：

- `代表结论卡`：一句结论、验收状态、协议阶段、展示图包、对应计划、下一实验
- `展开实验结论导读`：需要时再看完整 7 项说明和结果浏览建议

## 3. 页面之间的推荐顺序

### 第一次使用平台

1. 平台总览
2. 实验库
3. 创建实验
4. 结果总览

### 已经有计划，继续推进

1. 计划管理
2. 当前计划
3. 保存并运行
4. 结果总览
5. 结果对比 / 姿态回放

### 已经有结果，继续分析

1. 结果总览
2. 当前结果
3. 结果对比
4. 姿态回放
5. 回到计划管理或实验库

## 4. 当前界面的核心判断逻辑

### 计划层

先回答：

- 现在要继续哪一个计划
- 这个计划有没有结果
- 它属于哪条实验主线
- 下一步是编辑、运行还是归档

### 结果层

先回答：

- 现在最该看哪一个结果
- 有没有失败或边界结果要优先处理
- 先读摘要、先看图，还是先回计划

### 回放层

先回答：

- 当前结果有没有值得看的动态过程
- 是否包含任务模式切换
- 是否包含可联动的 runtime / mission 信息

## 5. 与成熟平台范式的对应

当前 UI 不是单纯展示页面，而是平台层的一部分。它对应成熟卫星仿真平台中的几类职责：

| 成熟范式 | satmodel 对应 |
| --- | --- |
| Project / Workspace | 平台总览、计划管理 |
| Scenario / Template | 实验库、创建实验 |
| Experiment Plan | 创建实验、当前计划 |
| Result Browser | 结果总览、Dashboard 预览 |
| Analysis / Comparison | 结果对比 |
| Replay / Timeline | 姿态回放 |

这意味着 UI 设计原则应持续保持：

1. 先给平台视角，再给文件视角。
2. 先给实验主线，再给字段细节。
3. 先给判断入口，再给完整列表。

## 6. 当前边界

当前界面已经适合本地研究和展示，但还不是最终形态。

目前边界包括：

- 重点仍是本地单用户工作台
- 结果管理仍基于文件目录，而不是数据库
- 三维回放仍是轻量结果回放，不是高保真任务场景
- UI 已经平台化，但还可以继续减少说明文本、增加更强的图形化导航

## 7. 启动方式

### CLI

```bash
satmodel-platform-ui --root . --host 127.0.0.1 --port 8765
```

### 脚本

```bash
python scripts/serve_platform_ui.py --root . --host 127.0.0.1 --port 8765
```

### Windows 快捷入口

直接双击：

```text
打开平台界面.bat
```

默认地址：

```text
http://127.0.0.1:8765
```

## 8. 后续优化方向

下一阶段界面优化应继续围绕三件事推进：

1. 让首屏更短：
   - 减少大段说明文字
   - 更多用流程图、状态卡和入口卡
2. 让实验更直观：
   - 明确实验问题、变量、验收和推荐图包
   - 让计划页和结果页共享同一条实验主线
3. 让结果更适合展示：
   - 增强关键图入口
   - 增强最差 run 诊断和动态回放联动
   - 增强代表结果与代表计划绑定
